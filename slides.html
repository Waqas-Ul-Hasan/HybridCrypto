INTRODUCTION

The Problem: Secure File Transfer

🌐

Data in Transit

Files sent over networks can be intercepted by attackers using packet sniffing tools

💾

Data at Rest

Files stored on disk are vulnerable if a system is compromised or storage is stolen

🔓

Key Distribution

Symmetric encryption alone can't securely share the secret key between parties

Solution: Combine symmetric speed with asymmetric security — Hybrid Cryptography

THEORY

Why Hybrid Cryptography?

AES (Symmetric)

✅ Extremely fast
✅ Handles any file size
✅ Low CPU overhead
❌ Key distribution problem
❌ Both parties need same key

RSA (Asymmetric)

✅ Solves key distribution
✅ Public key freely shared
✅ Private key never transmitted
❌ Slow — can't encrypt files
❌ Limited data size (~214 bytes)

METHODOLOGY

Cryptographic Algorithms

Algorithm Type Key Size Mode/Variant Purpose
AES Symmetric 256-bit CBC + PKCS7 Encrypt file content (fast)
RSA Asymmetric 2048-bit OAEP + SHA-256 Encrypt AES session key
HMAC MAC / Integrity 256-bit tag SHA-256 Detect tampering
IV Randomness 128-bit CSPRNG Prevent pattern leakage

OAEP Padding

Optimal Asymmetric Encryption Padding adds randomness to RSA, preventing deterministic outputs and chosen-ciphertext attacks

Encrypt-then-MAC

HMAC is computed over ciphertext after encryption — provably secure ordering that detects tampering before decryption begins

IMPLEMENTATION

System Architecture

🔒 Encryption Flow

1. Generate random AES-256 key + 128-bit IV
2. Encrypt file with AES-256-CBC → ciphertext
3. Compute HMAC-SHA256(IV ∥ ciphertext)
4. Encrypt AES key with RSA public key (OAEP)
5. Bundle all into .enc file

.enc File Format

┌─────────────────────────────┐ │ MAGIC HEADER (8 bytes) │ ├─────────────────────────────┤ │ KEY LENGTH (2 bytes) │ ├─────────────────────────────┤ │ RSA-ENCRYPTED AES KEY │ │ (256 bytes for 2048-bit) │ ├─────────────────────────────┤ │ AES IV (16 bytes) │ ├─────────────────────────────┤ │ HMAC-SHA256 (32 bytes) │ ├─────────────────────────────┤ │ CIPHERTEXT (variable) │ └─────────────────────────────┘

CODE

Core Implementation

# AES-256-CBC Encryption from Crypto.Cipher import AES, PKCS1_OAEP from Crypto.PublicKey import RSA from Crypto.Random import get_random_bytes # 1. Generate session key aes_key = get_random_bytes(32) # 256-bit iv = get_random_bytes(16) # 128-bit IV # 2. Encrypt file data cipher = AES.new(aes_key, AES.MODE_CBC, iv) ciphertext = cipher.encrypt(pad(data, 16)) # 3. HMAC integrity tag mac = hmac.new(aes_key, iv+ciphertext, hashlib.sha256).digest() # 4. Encrypt AES key with RSA rsa_key = RSA.import_key(pub_pem) cipher_rsa = PKCS1_OAEP.new(rsa_key) enc_key = cipher_rsa.encrypt(aes_key)

DEMO

Live Demonstration

🔑

Step 1: Key Gen

python cli.py keygen \ --private keys/private.pem \ --public keys/public.pem

Generates RSA-2048 key pair with SHA-256 fingerprint

🔒

Step 2: Encrypt

python cli.py encrypt \ --input secret.txt \ --output secret.enc \ --key keys/public.pem

Produces encrypted .enc bundle unreadable without private key

🔓

Step 3: Decrypt

python cli.py decrypt \ --input secret.enc \ --output recovered.txt \ --key keys/private.pem

HMAC verified first; plaintext restored if integrity OK

Tamper Detection Demo: Flip bits in .enc file → INTEGRITY CHECK FAILED → Decryption aborted

SECURITY

Security Analysis

Strengths

Brute-force resistance (AES-256)MAX
Key exchange security (RSA-2048)HIGH
Tamper detection (HMAC-SHA256)MAX
Padding oracle resistance (OAEP)HIGH
Pattern concealment (random IV)MAX

Known Limitations

• Private key stored unencrypted on disk
• No perfect forward secrecy (RSA reuse)
• No key revocation mechanism
• Single-recipient encryption only

Future Improvements

• Passphrase-protected private key (AES-256-GCM wrapping)
• Digital signatures (RSA + SHA-256)
• Elliptic Curve Cryptography (ECDH)
• Multi-recipient support
