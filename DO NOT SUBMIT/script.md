### 🎙️ SLIDE 3 — What is Hybrid Cryptography? (1:00–2:00)

> *[Slide: Venn diagram or comparison table]*

**"Let me explain why we use a hybrid approach."**

**"Symmetric encryption — like AES — is extremely fast and operates with very low CPU overhead. It can efficiently encrypt gigabytes of data of any file size. However, it suffers from a critical key distribution problem: both the sender and receiver need to possess the exact same secret key."**

**"Asymmetric encryption — like RSA — solves this key distribution issue elegantly. You publish your public key freely, while your private key is never transmitted. Anyone can encrypt with the public key, but only the private key can decrypt. However, RSA is computationally heavy, extremely slow, and is limited to encrypting tiny amounts of data—usually around 214 bytes."**

**"The solution? Combine them:"**
- **AES encrypts the file** — fast, efficient
- **RSA encrypts only the AES key** — secure key exchange
- **Result: Best of both worlds**"**

---

### 🎙️ SLIDE 4 — Technical Algorithms (2:00–3:00)

> *[Slide: Algorithm specs table]*

**"Our system relies on four core cryptographic mechanisms:"**

**"First, we use AES-256 in CBC mode with PKCS7 padding to rapidly encrypt the actual file content, while hiding data patterns."**

**"Second, we use RSA-2048 based on the mathematical difficulty of factoring large prime numbers to securely encrypt that AES session key."**

**"Third, we employ HMAC-SHA256 to generate an integrity tag that detects any tampering."**

**"And fourth, we use a 128-bit Initialization Vector generated by a secure random number generator to prevent any pattern leakage in the ciphertext."**

**"Two critical security decisions tie this together: we use OAEP padding for RSA to prevent padding oracle attacks, and we follow the Encrypt-then-MAC paradigm to ensure tampering is detected before decryption even begins."**

---

### 🎙️ SLIDE 5 — System Architecture & Methodology (3:00–4:00)

> *[Slide: Encryption flow diagram]*

**"Here is our encryption methodology:"**

**"When you encrypt a file:**
1. We generate a **random 256-bit AES key** and a **random 16-byte IV** — unique for each encryption
2. The file is **encrypted with AES-256-CBC** producing the ciphertext
3. We compute **HMAC-SHA256** over the IV and ciphertext — creating an integrity tag
4. The AES key is **encrypted with the recipient's RSA public key** using OAEP
5. Everything is **bundled into a .enc file**"**

**"This `.enc` file is highly structured: it begins with an 8-byte magic header and key length indicator, followed by the 256-byte RSA-encrypted key, the 16-byte IV, the 32-byte HMAC tag, and finally the variable-length ciphertext."**

**"This entire architecture is implemented cleanly across our `hybrid_crypto.py`, `cli.py`, and `gui.py` project files."**

---

### 🎙️ SLIDE 6 — Core Implementation (4:00–4:30)

> *[Slide: Core Implementation code block]*

**"Here you can see the core Python implementation using the PyCryptodome library."**

**"We generate the 256-bit AES key and 128-bit IV using a Cryptographically Secure Pseudorandom Number Generator. The file is encrypted using AES in CBC mode with PKCS7 padding. We then generate the HMAC-SHA256 tag over the IV and ciphertext. Finally, the AES session key is encrypted using the recipient's RSA public key with OAEP padding."**

### 🎙️ DEMO 5 — Tamper Detection (6:30–7:10)

> *[This is the most impressive demo — shows security]*

**"Now let me demonstrate one of the most important security features — tamper detection."**

**"I'll simulate an attacker modifying the encrypted file."**

```python

python -c "
f = open('../samples/secret.enc', 'r+b')
f.seek(-10, 2)

f.seek(-10, 2)
f.write(bytes(b ^ 0xFF for b in data))

```

**"Now if we try to decrypt the tampered file..."**

```bash

```

**"The system immediately detects the tampering! The HMAC verification fails and decryption is aborted. The error message says: INTEGRITY CHECK FAILED. This is the Encrypt-then-MAC pattern in action."**

### 🎙️ SLIDE 10 — Security Analysis (8:10–8:45)

> *[Slide: Security analysis table]*

**"Analyzing the security strengths of our system, we achieve maximum resistance against brute-force attacks using AES-256, maximum pattern concealment using random IVs, and maximum tamper detection using HMAC-SHA256."**

**"Our key exchange security is highly robust thanks to RSA-2048, and we have high resistance to chosen-ciphertext and padding oracle attacks via OAEP."**

**"However, we also acknowledge some known limitations. The private key is currently stored unencrypted on disk, and there is no perfect forward secrecy. Future improvements would include wrapping the private key with an AES-256-GCM passphrase, adding digital signatures, and migrating to Elliptic Curve Cryptography."**
